# 🚨 Fix first — the critical list

Ordered by severity. Items 1–4 must be fixed **before any public deployment**; each is independently exploitable by an anonymous visitor.

1. **Fake admin auth: the password ships in the public JS bundle** — `AdminClient.tsx:137` uses `NEXT_PUBLIC_ADMIN_PASSWORD || "admin123"` compared client-side, and the login placeholder (line 407) literally advertises the default password. There is no server-side auth anywhere. → Replace with real auth (Supabase Auth + admin role, or server-verified session) and move all admin writes behind server routes using the service-role key. *(Auth section)*

2. **RLS lets anonymous users rewrite the whole database** — `supabase_setup.sql:29-30` grants public UPDATE on *all* columns of `prompts` (policies can't restrict columns, despite the policy's name), and `supabase_admin_policies.sql` "Option A" — which the admin panel currently *requires* to function — adds public INSERT/DELETE on prompts & news **and public SELECT/DELETE on the subscriber email list (PII breach)**. → Drop these policies; do counter increments via SECURITY DEFINER RPCs; do admin CRUD server-side. *(Security section)*

3. **Newsletter signups are written to a local JSON file, not Supabase** — `api/subscribe/route.ts:5,45`. On Vercel/serverless every subscription errors or is silently lost on cold start, and `src/data/subscribers.json` is not gitignored (PII can end up in git history). The Supabase `subscribers` table already exists with a safe insert-only policy — use it. *(Payment/webhook → Newsletter section)*

4. **XSS in the Explore prompt sandbox** — `ExploreClient.tsx:256-257 + 788`: user-typed variable values are spliced unescaped into `dangerouslySetInnerHTML`. Self-XSS today, stored XSS the moment those values persist. → Escape the values or render React nodes instead of HTML strings. *(Security section)*

5. **Supabase errors are silently swallowed on every counter write** — the client returns `{ error }` and never throws, so the `try/catch` blocks in HomeClient/ExploreClient around `views`/`likes` updates catch nothing and no failure is ever surfaced or logged. Related: likes/views are written as absolute values from stale client state (lost updates), and bookmarking mutates the global like counter. → Check `error`, use atomic RPC increments, decouple bookmarks from likes. *(Components section)*

6. **Admin UI lies about persistence** — on a failed DB write/delete, `AdminClient.tsx` still updates the local table (delete even runs unconditionally), so an admin can "save" content that vanishes on refresh. Editing also clobbers live view/like counters with stale snapshots. *(Data integrity section)*

7. **Production deploys silently fall back to fake data** — `src/lib/supabase.ts:7-8` builds a client against a placeholder URL when env vars are missing; the site then serves the bundled 15 mock prompts and a fake "sandbox login" that accepts any credentials. → Fail fast in production; export a nullable client. *(Error handling section)*

8. **The content site is invisible to search engines** — all content renders client-side after a browser Supabase fetch; prompts/news have no per-item routes (slugs are dead); `/og-image.jpg` and `/previews/*.jpg` don't exist; no sitemap/robots; `/admin` is indexable and describes itself as the admin panel. → Server-render data, add `[slug]` routes, `robots.ts` + `sitemap.ts`, noindex admin. *(Pages & routes section)*

9. **Schema/type drift across three sources of truth** — SQL schema (snake_case, no `image_url`, nullable `description`), `types.ts` (camelCase, required fields), and hand-written per-component mappers have already diverged; content data is duplicated between `mockData.ts` and the SQL seed. → Generate DB types (`supabase gen types typescript`), single content source. *(Data integrity section)*

10. **Compliance & accessibility gaps to close before launch** — Terms/Privacy links are placeholders while the site collects emails and passwords; footer claims MIT license but no LICENSE file exists; hero claims "150+ prompts" (there are 15); `* { outline: none }` removes all keyboard focus indicators; `--accent-red` is used but never defined (delete buttons aren't red); the /explore sidebar never collapses on tablets. *(Components / Styling sections)*

**Also notable:** duplicated ~600 lines between HomeClient and ExploreClient (fixes must be applied twice — e.g. the two pages already use *different* localStorage keys for newsletter status); no tests or typecheck script; README is stock boilerplate with no setup instructions and no `.env.example`.

## Security

### Area: Supabase SQL & RLS

- 🔴 **CRITICAL — supabase_admin_policies.sql:14-26 ("Option A")** — This script makes the entire database world-writable with the *public anon key*: anyone can INSERT/UPDATE/DELETE any prompt or news row (lines 15-22), DELETE any subscriber (line 25), and — worst — **SELECT the full subscriber email list (line 26)**, which is a PII breach. *Why it matters:* the file's own header says to run it "if you encounter permission issues when saving edits from /admin" — i.e., it is the required path for the admin panel to function, because the admin panel is a client component using the anon key (confirmed in later areas). Any visitor who opens dev tools and copies the anon key from the page bundle can wipe or deface the whole site and harvest subscriber emails. *Fix:* do not run Option A anywhere except a throwaway local sandbox. Implement the admin as server-side API routes (or Server Actions) that verify an admin session and use `SUPABASE_SERVICE_ROLE_KEY` server-side (the file's "Option B" comment describes this but no such code exists in the repo — it must be built).
- 🔴 **CRITICAL — supabase_setup.sql:29-30** — `CREATE POLICY "Allow public update of views and likes" ON prompts FOR UPDATE USING (true) WITH CHECK (true)`. Despite the name, **RLS policies cannot restrict which columns are updated** — this grants anonymous users the right to update *every column of every prompt*: title, prompt_text, slug, category, all of it. Anyone with the anon key can deface all content even in the "secure" baseline setup. *Fix:* remove this policy. For view/like counters, create SECURITY DEFINER RPC functions, e.g. `create function increment_views(p_id text) ... update prompts set views = views + 1 where id = p_id;` and call them via `supabase.rpc(...)`. Alternatively use column-level privileges (`REVOKE UPDATE ... GRANT UPDATE (views, likes)`) combined with the policy — but the RPC is safer and also fixes the read-modify-write race (see Data integrity).
- ⚠️ **supabase_setup.sql:65-66** — `subscribers` INSERT policy is `WITH CHECK (true)` with no constraint on email format and no rate limiting. Bots can bulk-insert garbage rows directly against the Supabase REST endpoint (bypassing your API route entirely). *Fix:* add a CHECK constraint for basic email shape (`email ~* '^[^@\s]+@[^@\s]+\.[^@\s]+$'`), and consider captcha/rate limiting at the API layer plus keeping the anon INSERT but never trusting the table contents.

### Area: /src/components/ExploreClient.tsx

- ⚠️ **XSS — ExploreClient.tsx:256-257 + 788 — Sandbox variable values are injected into `dangerouslySetInnerHTML` unescaped.** `getHighlightedPreview` carefully HTML-escapes the prompt text (lines 241-246) but then splices the raw user-typed `userVal` into the markup: `` escaped.replace(bracketRegex, `<span ...>${userVal}</span>`) ``. Typing `<img src=x onerror=alert(1)>` into a "Prompt Sandbox" variable input executes script in the page. Today the value never leaves the visitor's browser (self-XSS), so severity is moderate — but it is one persistence feature away from stored XSS, and `userVal` containing `$&`/`` $` `` also corrupts output via `String.replace` special patterns. *Fix:* escape `userVal` with the same entity map before insertion (and escape `$` as `$$` or use a replacer function); better, drop `dangerouslySetInnerHTML` and render an array of text/`<span>` React nodes.

## Data integrity / Supabase & RLS

### Area: Supabase SQL & RLS (supabase_setup.sql, supabase_admin_policies.sql — both read in full)

- **supabase_setup.sql:16-17** — `views INTEGER DEFAULT 0, likes INTEGER DEFAULT 0` are nullable and have no `CHECK (>= 0)`. Combined with the public UPDATE policy, counters can be set to NULL, negative, or 999999999 by anyone. *Fix:* `NOT NULL DEFAULT 0 CHECK (views >= 0)`, and make increments atomic via RPC (see Security). The client-side read-modify-write increment (confirmed in the components area) also loses counts under concurrency even for honest users.
- **supabase_setup.sql:8, 35** — `id TEXT PRIMARY KEY` with hand-assigned ids (`'p1'`, `'n1'`). Client code must invent ids, which invites collisions and makes inserts racy. *Fix:* `id UUID PRIMARY KEY DEFAULT gen_random_uuid()` (as already done for `subscribers`), or at least generate ULIDs server-side.
- **supabase_setup.sql:19, 39** — `date TEXT NOT NULL -- 'YYYY-MM-DD'` stores dates as strings. No validation: `'not-a-date'` inserts fine; sorting only works by accident of ISO format. *Fix:* use the `DATE` type; Supabase returns it as an ISO string to JS anyway, so client changes are minimal.
- **supabase_setup.sql:12, 18, 42** — `category`, `difficulty`, `importance` are free TEXT with allowed values only in comments. Nothing stops `difficulty = 'banana'`, which would silently fall through every filter in the UI. *Fix:* add CHECK constraints or Postgres enums (`CHECK (difficulty IN ('Beginner','Intermediate','Advanced'))`, etc.).
- **supabase_setup.sql:57** — `subscribers.email` is `TEXT UNIQUE` but not normalized: `Foo@Bar.com` and `foo@bar.com` insert as two rows. *Fix:* use `citext`, or a unique index on `lower(email)`, and lowercase in the API route before insert.
- **supabase_setup.sql:78-185** — Seed INSERTs have no `ON CONFLICT DO NOTHING`. The tables use `CREATE TABLE IF NOT EXISTS`, implying the script is meant to be re-runnable, but a second run aborts with duplicate-key errors halfway. *Fix:* append `ON CONFLICT (id) DO NOTHING`.
- **prompts/news tables** — No `created_at`/`updated_at` timestamps. The TEXT `date` field is editorial, not an audit trail; after any tampering incident (very possible given the policies above) you'd have no forensics. *Fix:* add `created_at timestamptz default now()`, `updated_at` via trigger.
- ✅ **supabase_setup.sql:69-70** — The explicit `FOR SELECT USING (false)` deny policy on subscribers is redundant (no policy = deny) but harmless; documents intent. Fine.
- ✅ RLS is at least *enabled* on all three tables — the baseline read policies (public SELECT on prompts/news) are appropriate for a public content site.

### Area: /src/lib + /src/types + /src/data

- **src/types.ts vs supabase_setup.sql — schema/type mismatch (multiple fields)** — TS uses `targetAI`, `promptText`, `outputDescription`, `sourceUrl`, `imageUrl` (camelCase); the DB columns are `target_ai`, `prompt_text`, `output_description`, `source_url`, and **`image_url` does not exist in the DB at all**. Supabase returns snake_case, so every fetch requires a hand-written mapping layer (done ad hoc inside each component — see Components area), and any prompt saved from the admin panel can never persist an image. *Why it matters:* three sources of truth (SQL schema, TS types, per-component mappers) that have already diverged; a renamed column breaks silently at runtime, not compile time. *Fix:* generate types with `supabase gen types typescript`, use `createClient<Database>`, and add the missing `image_url` column or drop `imageUrl` from the type.
- **src/types.ts:5,9 vs supabase_setup.sql:11,15** — `description` and `outputDescription` are required (non-optional) strings in TS but nullable in the DB. A row inserted with NULL (possible via the public policies) type-checks as `string` and then crashes things like `.substring()`/`.length` at runtime. *Fix:* align — either `NOT NULL DEFAULT ''` in SQL or optional in TS.
- **src/data/mockData.ts (entire file) vs supabase_setup.sql:78-185** — The 15 prompts + 4 news items are fully duplicated between the SQL seed and this TS file, and they have *already drifted*: mockData gives p1–p5 an `imageUrl`, the SQL seed has no such column. *Why it matters:* the app silently swaps between DB data and mock data (see supabase.ts finding below), so users can see different content depending on env configuration, and every content edit must be made twice. *Fix:* generate one from the other, or drop the SQL seed and provide a small seeding script that imports mockData.
- **src/data/mockData.ts:25,47,62,82,97** — `imageUrl: "/previews/*.jpg"` for p1–p5, but `public/` contains only the 5 default SVGs — **no `previews/` folder exists**. Any component rendering these gets a 404/broken image. *Fix:* add the images or remove the fields.

### Area: /src/app/admin/AdminClient.tsx

- **AdminClient.tsx:200-234 & 240-252 — Local state diverges from the database on write failure.** In `handlePromptSave`, if the Supabase upsert throws, execution falls through and the item is still added/updated in local React state (lines 229-233); the admin sees it in the table as if saved, and it vanishes on refresh. `handlePromptDelete` is worse: the DB delete can fail (line 247 toast) yet line 252 removes the row from the UI unconditionally. Same pattern for news (286-333, 335-351) and subscribers (355-369). *Fix:* only mutate local state after a confirmed DB success (or re-fetch after each mutation); on failure keep the UI truthful.
- **AdminClient.tsx:182-183,194-195 — Editing a prompt overwrites live counters with stale values.** The form snapshots `views`/`likes` when the modal opened and writes them back on save, clobbering any likes/views accrued in between. *Fix:* omit `views`/`likes` from the update payload entirely (use `.update()` with only editable columns instead of a full-row upsert).
- **AdminClient.tsx:666-676 — The ID field is user-editable on create, and saves use `upsert(..., { onConflict: "id" })`.** Typing an existing id silently *replaces* that prompt instead of erroring. Same for news (line 800-808). *Fix:* ids shouldn't be user input at all (see the DB `gen_random_uuid()` recommendation); use plain `insert` for creates so collisions fail loudly.
- **AdminClient.tsx:683-686,812-818 — Slug is not validated or checked for uniqueness.** The DB has `UNIQUE NOT NULL` on slug, so a duplicate slug makes the upsert fail with only a generic "Database insert failed" toast; nothing stops spaces/uppercase/empty-after-trim slugs either. *Fix:* validate format (`/^[a-z0-9-]+$/`), and surface the actual Supabase error message in the toast.

## Auth

### Area: /src/app/admin/AdminClient.tsx (read in full, 910 lines)

- 🔴 **CRITICAL — AdminClient.tsx:137 — Admin "password" is client-side and shipped to every visitor.** `const envPassword = process.env.NEXT_PUBLIC_ADMIN_PASSWORD || "admin123";` — any `NEXT_PUBLIC_*` variable is **inlined in plaintext into the public JS bundle** at build time; anyone can read the real password from the served JavaScript. The comparison runs in the browser and "authentication" is just a React `useState` boolean (line 15), so it can also be flipped in React DevTools. There is no session, no cookie, no server-side check anywhere. *Fix:* real auth — Supabase Auth with an admin role claim checked in RLS policies, or at minimum a server-verified session (middleware/proxy) + service-role writes behind server routes. A client component can never gate anything.
- 🔴 **AdminClient.tsx:407 — The login screen publishes the default password.** The password input's placeholder is literally `"default: admin123"`. Combined with the fallback on line 137, every deployment that forgot to set the env var has a publicly advertised admin login. *Fix:* remove the fallback AND the placeholder; fail closed if no auth is configured.
- **AdminClient.tsx:135-145 — No throttling/lockout on login attempts** (moot once the above is fixed properly, but worth stating: client-side string comparison is brute-forceable at CPU speed and irrelevant anyway since the expected value ships in the bundle).
- **AdminClient.tsx:438 — "Log Out" just sets local state to false.** Nothing is invalidated; back-button or DevTools restores access. Consistent with the fake auth model — fixed by the same rework.
- **Cross-reference (Security):** AdminClient.tsx:104 fetches the **subscribers table from the browser with the anon key**, and lines 202-206/242/304-306/340/360 do all CRUD the same way. This only works if the world-open "Option A" policies from supabase_admin_policies.sql are applied — i.e., the admin panel's design *forces* the critical RLS misconfiguration documented in the Security section. The panel must move to server-side routes with the service-role key.

## Newsletter subscribe route (/src/app/api/subscribe/route.ts — read in full)

- 🔴 **route.ts:5,45 — Subscriptions are written to a local JSON file, not Supabase.** The whole `subscribers` table + RLS policies in supabase_setup.sql are dead infrastructure; the route writes `src/data/subscribers.json` instead. *Why it matters:* (1) On Vercel/serverless — the deploy target the README assumes — the filesystem is ephemeral and often read-only: every subscription either throws `EROFS` (500 for the user) or is **silently lost on the next cold start**. (2) It writes PII into the source tree, and `src/data/subscribers.json` is **not in .gitignore** — one careless `git add .` in dev commits subscriber emails to history. *Fix:* insert into the Supabase `subscribers` table (`supabase.from('subscribers').insert({ email })`), which already exists with a correct insert-only RLS policy; delete the fs logic.
- **route.ts:24-45 — Read-modify-write race condition.** Two concurrent POSTs both read the file, both append, and the second write clobbers the first — a subscriber is silently lost. Also `readFileSync`/`writeFileSync` block the event loop. Moot once the route uses the DB (unique constraint handles duplicates atomically).
- **route.ts:9-16 — Input not validated as a string.** `const { email } = await request.json()` — if a client POSTs `{"email": 123}` or `{"email": {"a":1}}`, `email.includes` / `email.trim()` throws and the user gets a 500 instead of a 400. Also `request.json()` on a non-JSON body throws into the same generic 500. *Fix:* `if (typeof email !== "string") return 400;` and wrap the body parse separately.
- **route.ts:11 — Validation is only `email.includes("@")`.** `"@"`, `"a@"`, a 10 MB string — all accepted and stored verbatim. *Fix:* a minimal regex (`/^[^\s@]+@[^\s@]+\.[^\s@]+$/`) plus a length cap (e.g. 254 chars), after trimming/lowercasing *before* validating (currently it validates the raw value but stores the normalized one).
- **route.ts:36-41 — Subscriber enumeration.** The distinct "This email is already subscribed!" response lets anyone probe which emails are on the list. Low severity for a newsletter, but the standard fix is to return the same success message for duplicates.
- **No rate limiting / abuse control** — nothing stops a bot from posting thousands of emails. *Fix:* at minimum, add a same-IP throttle (e.g. Upstash ratelimit or Vercel firewall rule); pair with the DB unique constraint.
- **route.ts:51** — `catch (error: any)` — use `unknown`; `@typescript-eslint/no-explicit-any` will flag this once linting runs on it.
- ✅ Route handler signature (`export async function POST(request: Request)`) is valid Next 16 App Router convention; response shapes (`NextResponse.json` with status) are correct. Duplicate check and stored value are consistently normalized (trim+lowercase).

### Area: /src/app/admin/AdminClient.tsx (error handling)

- **AdminClient.tsx:637-648 — The "Resend Marketing List Sync" status card is dead/deceptive code.** `process.env.RESEND_API_KEY` (line 642) is a *server-only* variable — in a `"use client"` component it is always `undefined`, so the label always shows "MISSING" regardless of configuration; the badge on line 641 checks a *different* expression (`NEXT_PUBLIC_RESEND_CONNECTED === "true" || ...`), so badge and label can disagree. The card's text claims "submissions are saved to the Supabase backup table automatically" — false: the subscribe route writes a local JSON file and there is **no Resend integration anywhere in the repo**. *Fix:* delete the card, or build the integration server-side and report its status honestly.
- **AdminClient.tsx:113-115 — All three fetches share one try/catch**, so if e.g. only the `subscribers` SELECT fails (likely — its RLS read policy is deny-by-default unless Option A was run), the prompts/news that *did* load before the throw are kept but the toast claims "Operating in mock mode" while the seeded fake subscriber list (lines 22-25) is displayed as if real. *Fix:* per-table error handling; show an explicit error state instead of fake data.
- **AdminClient.tsx:22-25 — Hardcoded fake subscribers** (`onboarding@appprompthub.com`, …) render as real rows in the admin table whenever the DB fetch doesn't overwrite them. An admin can't distinguish mock from real data. *Fix:* start with `[]` and show an empty/error state.
- **AdminClient.tsx:127-132 — Toast timer isn't reset for back-to-back messages.** If a second toast fires while one is visible, `showToast` stays `true`, the effect doesn't re-run, and the old timer hides the new message early. *Fix:* key the effect on `toastMessage` too, or use a counter.
- **AdminClient.tsx:373-375 — `navigator.clipboard.writeText(...).then(...)` has no `.catch`.** Clipboard can reject (permissions, non-secure context); the rejection is unhandled and the user gets no feedback. *Fix:* add a catch with an error toast.

## Pages & routes

### Area: /src/app core routes (layout.tsx, page.tsx, explore/page.tsx, admin/page.tsx — all read in full)

- **Architecture: the entire site is three thin server pages wrapping giant client components.** `page.tsx`, `explore/page.tsx`, and `admin/page.tsx` render `<HomeClient/>`, `<ExploreClient/>`, `<AdminClient/>` with zero server-side data fetching. All prompt/news content is fetched in the browser with the anon key after hydration. *Why it matters:* (1) **SEO** — the site's whole premise is an indexable prompt library, but crawlers receive pages whose real content arrives via client-side Supabase calls; the SSR HTML contains only the mockData fallback (stale/divergent content — see Data integrity). (2) Every visitor pays a client-side fetch waterfall. *Fix:* fetch prompts/news in the server components (Next 16 supports async server components with fetch caching/revalidation per `01-getting-started/06-fetching-data.md`) and pass data down as props; keep interactivity in smaller client leaf components.
- **No per-prompt routes exist.** Prompts and news items have unique `slug` columns, but the app defines only `/`, `/explore`, `/admin` — items open in client-side modals (see Components area). Individual prompts have no URL, so they can't be indexed, shared, or deep-linked. The `slug` field is effectively dead. *Fix:* add `app/prompt/[slug]/page.tsx` (and `app/news/[slug]/page.tsx`) with `generateStaticParams` + `generateMetadata`.
- **layout.tsx:24,37 — `/og-image.jpg` does not exist** in `public/` (which holds only the create-next-app SVGs + favicon). All social shares render with a broken preview image. *Fix:* add the 1200×630 image or remove the references.
- **admin/page.tsx:3-6 — Admin page is indexable and advertises itself.** Metadata literally says "Secure panel to manage prompts…" with no `robots: { index: false }`, no sitemap exclusion, and no server-side auth gate on the route (auth is inspected in the AdminClient area — it is client-only). Search engines will index `/admin` and its description invites attention. *Fix:* add `robots: { index: false, follow: false }` to the metadata, and enforce real auth via proxy/middleware or server-side session check — not just in the client component.
- **No `error.tsx`, `not-found.tsx`, or `loading.tsx` anywhere in `app/`** (Next 16 file conventions confirmed in bundled docs: `error.md`, `not-found.md`, `loading.md`). Any render error surfaces the default Next error screen; unknown URLs get the unstyled default 404. *Fix:* add at least a root `error.tsx` and `not-found.tsx` matching the site theme.
- **No `robots.ts` / `sitemap.ts`.** For a content-library site this is a meaningful SEO gap, and `robots.ts` is also where `/admin` should be disallowed. *Fix:* add both (trivial in Next 16 via the metadata file conventions).
- **layout.tsx:47-48 — `suppressHydrationWarning` on both `<html>` and `<body>`.** There is no theme/dark-mode attribute toggling that would justify it (globals.css handles theme via media query at most). This flag silences real hydration mismatch warnings on those elements — likely added to paper over a mismatch caused by the client components' date formatting or similar (see Components). *Fix:* remove it, then fix whatever warning appears.
- ✅ **layout.tsx metadata block (lines 10-39)** — Well-formed: `metadataBase`, canonical, OpenGraph, Twitter card all correct per Next 16 metadata API. Good.
- ✅ **explore/page.tsx** — Fine as a wrapper; consider typing the metadata export `: Metadata` for consistency with layout.tsx (minor).
- ✅ **Font setup (layout.tsx:2-8)** — `next/font/google` with CSS variable; correct Next 16 pattern.

## Components

### Area: /src/components/HomeClient.tsx (read in full, 1,905 lines)

- 🔴 **HomeClient.tsx:220-227, 241-247, 395-401 — Supabase errors are silently swallowed on every counter write.** The Supabase JS client does **not throw** on query failure — it returns `{ error }`. These `try { await supabase.from("prompts").update(...) } catch` blocks never catch anything; the returned `error` is never checked. If RLS denies the update or the network fails, the UI shows the incremented number, the DB keeps the old one, and no log fires. *Fix:* `const { error } = await ...; if (error) ...` — everywhere the client is awaited without destructuring.
- 🔴 **HomeClient.tsx:222, 243, 397 — Likes/views are written as absolute values computed from client state** (`update({ likes: promptItem.likes + 1 })`). Two users clicking simultaneously → one increment lost; a user with a stale tab open overwrites hours of counts. This is also the feature forcing the dangerous public-UPDATE RLS policy (see Security). *Fix:* atomic `increment_likes(id)` / `increment_views(id)` SECURITY DEFINER RPCs.
- **HomeClient.tsx:202-257 — Bookmarking mutates the global like counter.** "Favorites" are per-browser localStorage, but toggling one increments/decrements the *sitewide* `likes` column. Un-bookmarking something months later decrements today's count; repeated toggling lets one user pump the counter up and down at will. *Fix:* decouple — either likes are their own action with per-user dedup (needs auth), or bookmarks stay purely local and don't touch `likes`.
- **HomeClient.tsx:128, 155 — `if (dbPrompts && dbPrompts.length > 0)`:** when the DB is configured but a table is legitimately *empty* (e.g. admin deleted everything), the UI silently keeps showing the 15 hardcoded mock prompts as if they were real. *Fix:* trust the DB result when credentials exist; reserve mock data for the no-credentials case.
- **HomeClient.tsx:142 — `imageUrl: p.imageUrl || undefined` is dead mapping:** the DB column would be `image_url` (snake_case) and doesn't exist anyway, so this is always `undefined`. Symptom of the missing generated types (see Data integrity).
- **HomeClient.tsx:405-434 — `handleNewsletterSubmit` is dead code** (verified: defined, never referenced — both newsletter forms use `handleSubscribe`). 30 lines to delete.
- **HomeClient.tsx:78-90 & 785-812 — Tab state is read from `?tab=` once on mount but never written back.** Switching tabs doesn't update the URL, so refresh/back-button/sharing all lose the tab; meanwhile the "builder", "news", and "workspace" experiences have no stable URL at all. *Fix:* make the tabs real routes (`/builder`, `/news`, `/workspace`) or at least sync `?tab=` via `router.replace` — this also fixes the SEO problem noted under Pages & routes.
- **HomeClient.tsx:1434-1491 — The workspace "auth lock" is cosmetic.** Bookmarks, custom templates, and copy history live only in localStorage; nothing is keyed to the signed-in account. Logging in on another device shows an empty workspace, and *not* logging in loses nothing (the data was already in the browser). In sandbox mode (no Supabase creds), lines 479-499 accept **any email/password** as a successful "login". *Why it matters:* users are asked to create real credentials for zero benefit, and the mock path can ship to production if env vars are missing (see supabase.ts finding). *Fix:* either persist workspace data per-user in Supabase (with owner-scoped RLS — the proper use of auth here), or drop the login gate.
- **HomeClient.tsx:438-457, 363-365 — `navigator.clipboard.writeText(...).then(...)` with no `.catch`** (three call sites incl. `handleCopyAsJSON`). On non-HTTPS or denied permission the promise rejects unhandled and the user gets no feedback.

## Error handling

### Area: /src/lib + /src/types + /src/data

- **src/lib/supabase.ts:7-8** — When env vars are missing, the client is created against `https://placeholder-project.supabase.co` with a fake key instead of failing fast. Any code path that forgets to check `hasSupabaseCredentials` fires real network requests at a nonexistent host (slow DNS failures, hung fetches). Worse, a *production deploy with missing env vars* doesn't crash — the site silently serves stale `mockData`, which looks like it's working. *Fix:* in production (`process.env.NODE_ENV === 'production'`), throw at module load if the vars are absent; keep the mock fallback for dev only, and prefer exporting `supabase: SupabaseClient | null` so TypeScript forces the credential check at every call site.
- **src/lib/supabase.ts:10-18** — The missing-credentials warning only fires in the browser (`typeof window !== 'undefined'`). Server-side consumers (the `/api/subscribe` route imports this same client) get no log at all, so a misconfigured server fails silently. *Fix:* log on the server too (a `console.warn` is fine there — it lands in server logs, not user consoles).
